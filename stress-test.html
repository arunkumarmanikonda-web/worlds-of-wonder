Architecture Truth — What Is Actually Being Tested
This is a 100% static website. There is no Node.js, no Express, no backend server, no database server, no API gateway. The only real HTTP endpoints are the platform's built-in Tables REST API. Here is what each "step" in the stress test actually does:
1
Park Selection
JS only
Random pick from PARKS config. No HTTP.
2
Date Picking
JS only
Random future date. No HTTP.
3
Price Calc
booking.js
Exact TICKET_CATEGORIES pricing + GST + group discount.
4
Offer Apply
JS only
Rate-based discount. No API call.
5
GST Invoice
booking.js
CGST 9% + SGST 9% split. Pure JS.
6
Payment
JS only
Simulated Razorpay. No real payment gateway call.
7
API Write
REAL HTTP
POST tables/stress_bookings → 201. This is the actual test.

What This Stress Test Actually Measures
✅ Tables REST API throughput — How many concurrent POST/PATCH requests the Tables API handles per second
✅ API write latency — End-to-end HTTP latency for each booking record write
✅ Booking JS engine accuracy — Pricing, GST, group discounts calculated correctly at scale
✅ Concurrency handling — Race conditions, duplicate booking_ref generation at high VU counts
✅ Error rate — API 4xx/5xx error frequency under load
✅ Retry resilience — Exponential backoff behavior on failures
⚠️ What this does NOT test:
❌ Real payment processing (Razorpay is simulated)
❌ Email/SMS delivery (no notification service)
❌ QR code generation pipeline (client-side only)
❌ Session/auth token validation under load (no backend auth)
❌ Database query performance (Tables API abstracts this)
